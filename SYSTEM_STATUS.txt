KEY ARCHITECTURAL INSIGHTS:
============================

1. TOOL REGISTRY PATTERN
   Every tool is defined as a ToolDefinition with:
   - name (string literal)
   - description
   - Zod schema for args + output
   - async handler(args, ctx)
   
   This auto-converts to OpenAI tool spec for NIM, ensuring
   type safety and consistency across the agent loop.

2. CANVAS EVENT STREAMING
   The agent loop emits { event: 'canvas:company_card', data: {...} }
   during execution. The client subscribes via useChat().data and
   animates each card/row as it arrives, creating a live-building
   effect without waiting for the full response.

3. CONTACT DISCOVERY INTENSITY LEVELS
   Users choose how aggressive the contact finder is:
   - low: verified-only (direct match, cross-source)
   - medium: + LinkedIn snippets (SerpAPI/Bing)
   - high: + pattern inference + phone extraction (all clearly labeled)
   
   Each layer has different confidence/precision tradeoffs.

4. MULTI-LLM FALLBACK CHAIN
   On NIM 429 (rate limit), automatically fail over to OpenRouter,
   then Groq. Circuit breaker pattern prevents cascading failures.
   All embeddings always use NIM (consistent vector space).

5. SEARCH STRATEGY (MVP FIRST, PHASE 2 UPGRADE)
   MVP uses PostgreSQL with:
   - pg_trgm for fuzzy (typo tolerance)
   - FTS for ranking (recency, relevance)
   - pgvector for semantic (embeddings)
   
   Phase 2 swaps in Meilisearch via the same SearchProvider interface
   without touching agent or UI code.
